CSC130 Project 2: Scientific Calculator

general outline

Implimentation!

1. My new idea is to build a scientific calculator that uses Javascript to run calculations behind an HTML/CSS interface.

2. The main idea is 2D, but as with my previous project, a 3d interface may be much more intuitive for a user.

3. I expect this app to be able to run calculations at a level equivalent to a SHARP EL-510R, the official calculator of UVIC's math department.

4. This idea is important because calculators are very important tools and everyone needs access to them.

5. I do not have plans for true XR for this project, however I believe that audio-visual feedback could be a fun addition to the app.

    6.
  • Calculators do not need to be in XR, they work fine on a screen.
  • However, XR interfaces are inherently much more intuitive than 2D interfaces.
  • XR apps offer larger 'desktop' space, which may be beneficial for complex math.
    7.
  • 3D interfaces can be easily implemented if the calculator is already designed to tackle 3D graphing.
  • Calculators are simplisic, single-window interfaces and do not require 3d.
  • a 3d interface for a simple app may feel intimidating.

8. in what ways are AR, MR, and VR useful for the project?

AR provides immersive interfaces that interact with a real-world environment. This is unnecessary as a calculator is a self-contained app.

MR provides a virtual interface that is overlayed in the real world. This can have the same effect as adding another monitor to a desktop setup, and is always handy.

VR places the user in an entirely virtual interface. This is rather unnecessary for something as benign as a calculator.


9: auditing your idea

Does my idea exist in 2D? Yes, my idea exists in 2D. It is a simplistic app by nature.

Does my idea exist in 3D? No, my idea does not exist in 3D. Such an interface design would likely be unnecessary for a calculator.

An advantage of a 3d interface is that a 3d calculator interface would work well with graphing on more than 2 dimensions, however this app is intended to be accessible rather than advanced.


10: useful design principles

Useful: This calculator should posess a full range of functionality up to and possibly including graphing.

Usable: Every feature will be clearly accessible on one page.

Desirable: The main desirable trait of this app is the modularity of its interface.

Valuable: visually impaired users need accessible interfaces.

Findable: it will be marketed as an "accessible and modular interface" to indicate its purpose and attract users who need such a feature.

Accessible: The color and font size of the interface can be adjusted to acommodate for visually impaired users.


11: empathy map

When using my app, the user...

Thinks/feels: Acommodated for. They feel like this was designed to accomodate for them, because they are visually impaired and can adjust color and size to fit their needs.

Does/says: enjoys the modularity of the interface, spends some time making their own preferred color scheme.

Hears: text-to-speech of the button they are hovering their cursor over.

Sees: an aesthetically pleasing UI.

Pains: setting up a particular UI could require some work.

Gains: the ability to perform calculations with a UI that suits their needs/wants.


JSON object mock-up: color scheme preset

  • name (string): the name of the preset. Must be unique.
  • date (string): the date of the preset's creation in YYYY-MM-DD format.
  • backgroundColor (RGBa): RGB value for the preset's background color.
  • borderColor (RGBa): RGB value for the preset's border color.
  • highlightColor (RGBa): RGB value for the preset's highlighted element color.

As an example, a completed object will look like this:

{"name":"White theme","date":"2022-07-12","backgroundColor":"rgba(255,255,255,1)","borderColor":"rgba(0,0,0,0)","highlightColor":"rgba(0,0,255,1)"}

This preset is called White theme. It was created on 2022-07-12. It sets the background color to white, the border and text color to black, and the highlight color to blue.
